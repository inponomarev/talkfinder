How to properly blame things for causing latency: an introduction to distributed tracing and Zipkin

Adrian Cole

This session will overview how to debug latency problems, using call graphs created by Zipkin.

Latency analysis is an act of blaming components for causing user perceptible delay. In today’s world of microservices, this can be tricky as requests can fan out across polyglot components and even data-centers. This session will overview how to debug latency problems, using call graphs created by Zipkin.

This talk will review concepts, such as how tracing logging and metrics fit together. It will introduce Zipkin including its tracing data model and architecture. It will show a little demo code in JavaScript and Java, but code expertise isn’t required. When you leave, you’ll at least know something about distributed tracing, and hopefully be on your way to blaming things for causing latency!
